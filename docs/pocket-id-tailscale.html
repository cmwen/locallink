Private application SSO with Pocket ID and Tailscale

Pocket ID authenticates users into internal applications. Tailscale remains the network gate and keeps its existing login provider.

The two independent gates

Existing IdP → Tailscale login → private tailnet access
                                      ↓
Internal app → Pocket ID passkey → application session

Tailscale ACLs decide whether a device can reach Pocket ID and an application. Pocket ID then supplies standard OIDC identity, groups, and a user session to that application. Pocket ID is not used to enroll the device into Tailscale, so it can stay private.

1. Configure and start Pocket ID

cp .env.example .env
openssl rand -base64 32

# .env
POCKET_ID_APP_URL=https://pocket-id-host.your-tailnet.ts.net
POCKET_ID_PORT=1411
POCKET_ID_ENCRYPTION_KEY=<generated-value>
POCKET_ID_TRUST_PROXY=false

docker compose --profile identity up -d pocket-id

Keep the encryption key outside version control and back it up with the Pocket ID data. The issuer URL is an identity boundary: choose the final Tailscale HTTPS hostname before registering clients.

2. Publish private HTTPS

Bind Pocket ID to loopback and use Tailscale Serve as the TLS endpoint. On a host Tailscale node, the basic form is:

tailscale serve 1411
tailscale serve status

In a container-sidecar deployment, route a dedicated Serve HTTPS port to the reverse-proxy listener for Pocket ID. Do not enable Funnel: the issuer is intended only for authenticated tailnet members.

3. Register every internal application

  1. Open Pocket ID through its private HTTPS URL and complete /setup.
  2. Create one OIDC client per application.
  3. Copy the application's exact callback URL from its documentation.
  4. Store that application's client secret in its local secret configuration.
  5. Configure scopes such as openid profile email groups when supported.

The user's browser must reach both the application and Pocket ID. The application backend must also reach Pocket ID for discovery, token exchange, and signing keys.

Applications without native OIDC

Place OAuth2 Proxy, Caddy Security, Tinyauth, or a compatible forward-auth component in front of the application. The proxy becomes the Pocket ID OIDC client and passes authenticated identity to the upstream service.

Operational checks

docker compose ps pocket-id
curl -fsS "$POCKET_ID_APP_URL/.well-known/openid-configuration"
tailscale serve status

Test a new browser session and a second tailnet user before relying on the setup. Pocket ID downtime blocks new application sessions, while Tailscale connectivity and already-issued application cookies have their own lifetimes.
