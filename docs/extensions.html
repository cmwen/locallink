Workspace capabilities

How LocalLink out-of-box extensions work

Extensions declare optional capabilities around the service topology. The LocalLink dashboard and local web server do not depend on them: an extension-free workspace can still declare services, resolve ports, inspect runtime state and logs, and run lifecycle actions.

Enabling an extension expresses desired capability; it does not mean that LocalLink has installed software, published ports, or created credentials. The dashboard and locallink extensions now separate available, declared, waiting for external installation, waiting for a user, waiting for configuration, installed, and healthy states.

The declaration model

locallink.extensions.yml is the committable capability inventory. LocalLink resolves exposed ports from .env, reports missing required variables as setup work, and displays each extension in Connections. Runtime services remain in docker-compose.yml or locallink.services.yml; private values currently remain in .env.

Declaration readiness only means required values are present. Lifecycle checks separately verify host tools, external authentication, and private routes. For Private Edge, installing Tailscale and joining a tailnet remain explicit user-owned steps; route generation and verification are LocalLink-owned automation targets.

extension declaration → readiness in dashboard
service declaration   → lifecycle and runtime status
.env                   → local ports, URLs, and secrets

Dashboard

The local control surface reads the workspace model, runtime probes, connection declarations, and resource state. Capability cards are evidence-backed status displays; they do not claim that switching a preference installed or launched external software.

Reverse proxy

Maps stable origins or edge listeners to application ports. Caddy is the default example. LocalLink records the capability and its service dependencies; the Caddyfile remains the routing source of truth.

Tailscale private edge

Provides authenticated network reachability and private HTTPS. Tailscale Serve exposes only the routes explicitly declared in its configuration, and Tailscale ACLs remain the network authorization source. LocalLink never treats edge enablement as permission to make a service public.

locallink extension plan private-edge previews the complete onboarding boundary without modifying the workspace. Add quoted service names or IDs, for example locallink extension plan private-edge "My API", to preview an explicit exposure selection. The plan derives workspace-specific HTTPS listeners, checks the live Serve configuration for collisions, and includes exact apply and rollback commands. locallink extension apply private-edge "My API" records that selection alongside the workspace declaration and local Tailscale preference. After applying that workspace plan and reviewing a fresh live preview, locallink extension apply-routes private-edge "private-edge:<token>" publishes only those selected routes.

The route confirmation token binds approval to the exact workspace and generated command set. LocalLink rechecks it immediately before mutation, runs Tailscale without shell interpolation, verifies every selected listener, and records only routes it created in .locallink/workspace-state.json. If a command or verification fails, it rolls back the routes created by that attempt and reports any rollback failure for manual recovery. It never runs tailscale up, changes ACLs, enables Funnel, or replaces a conflicting listener.

When a selected service is later removed or its generated listener changes, the fresh plan includes an owned-route reconciliation section. locallink extension reconcile-routes private-edge "private-edge-removal:<token>" removes a listener only when its live target still matches the LocalLink ownership record. Already absent listeners and listeners changed by another actor are never deleted; LocalLink only forgets its stale local claim. A failed multi-route reconciliation restores listeners removed earlier in that same attempt when their ports remain safe to reclaim.

The route backend is an explicit adapter contract. adapter: tailscale-serve supports confirmed apply and reconciliation. adapter: tailscale-caddy supports Docker Caddy services that mount /etc/caddy/Caddyfile from inside the active workspace. LocalLink validates generated configuration, persists the original file and runtime ownership under .locallink, starts the Compose service when required, writes only a marked managed route block, reloads Caddy, reconciles deselected routes, restores the original file, and stops the service only when LocalLink started it. Apply and reconciliation failures restore the prior Caddyfile and running state. A host caddy command may be reported for visibility, but it is intentionally never treated as an automated runtime. Adapter identity travels with plans, tokens, and ownership records, and unknown adapters fail before LocalLink creates host commands.

The user must still join or create a tailnet, enable HTTPS certificates when required, provide a bootstrap credential, approve the access policy, and explicitly select which services may be exposed. Route generation stays blocked until that selection exists. LocalLink uses a stable workspace-derived listener range by default; set LOCALLINK_PRIVATE_EDGE_PORT_START only when a workspace needs pinned listener ports. Existing listeners are never replaced by planning. LocalLink owns verification and is the intended owner of generated proxy routes and derived URLs.

Pocket ID

Provides passkey-backed OIDC sessions to internal applications after a user reaches them through Tailscale. It depends on a stable HTTPS issuer, persistent data, and an encryption key. LocalLink can automate provider installation, issuer discovery, generated secrets, and health checks; the first administrator/passkey and application identity choices remain user actions. See the private SSO guide.

Applications should declare a provider-neutral OIDC contract including callback URLs, logout URLs, scopes, and environment-variable names. Coding agents can then configure the application without coupling it directly to Pocket ID.

OpenObserve

Receives telemetry from declared services. The target experience is for LocalLink to generate and store backend credentials, derive OTLP endpoints and authorization values, configure compatible exporters, and verify delivery.

Applications should use standard OpenTelemetry variables and protocols rather than OpenObserve-specific application code. Custom instrumentation may still require development work, but changing the telemetry backend should not require rewriting the service.

Coding-agent contract

LocalLink plans to provide an installable agent skill for discovering workspace capabilities, declaring services and Dockerfile blueprints, and integrating generic OIDC and OpenTelemetry interfaces. The skill should teach safe patterns; LocalLink remains the authority for generated configuration, secrets, and lifecycle state.

See the product goal for the agreed capability boundaries and configuration-ownership direction.

Default safety properties

  • No extension writes secrets into committed files.
  • No network edge is assumed public.
  • Only explicitly selected workspace ports can be matched to Tailscale routes or shown as edge URLs.
  • Missing required variables produce setup status rather than fake readiness.
  • Capability declarations do not replace Docker, PM2, Taskfile, proxy, or Tailscale configuration.
  • The dashboard does not expose a toggle that pretends to install or stop an external runtime.
  • Derived URLs, encoded headers, proxy routes, and generated credentials should not become duplicated human-maintained inputs.
