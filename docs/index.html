Local-first orchestration

One local control plane for Docker, PM2, PWA workflows, AI tooling, and workspace scaffolding.

LocalLink manages hybrid local runtimes, exposes an MCP toolkit for coding agents, serves an installable dashboard, scaffolds a version-controllable workspace, and now includes a Phase 2 advisor plus a live Resource watch for abnormal process detection.

PWA dashboard PM2 + Docker orchestration Workspace generator MCP toolkit Phase 2 advisor Resource watch

Architecture

LocalLink is intentionally small: the Node.js control plane reads workspace configuration, probes external runtime managers, and serves a local dashboard from the same loopback-only process. Docker, PM2, Taskfile, and the host process table remain the source of runtime truth.

Control plane src/app-context.ts composes config loading, runtime snapshots, lifecycle execution, port allocation, diagnostics, logs, and the HTTP/MCP entrypoints.
Dashboard surface public/ contains the framework-free PWA shell: launcher, dashboard, template preview, shared CSS, shared JS, manifest, service worker, and mock state.
MCP stdio server src/mcp/server.ts registers PRD-aligned tools so coding agents can inspect blueprints, allocate ports, verify compliance, and orchestrate declared services.
Workspace generator src/init/scaffold.ts creates the private infrastructure workspace with env files, Docker Compose, PM2, Taskfile, MCP registry, README, and AGENTS instructions.

Runtime state model

Every dashboard refresh asks Docker, PM2, Windows process probes, port scanning, diagnostics, and resource watchers for a fresh picture. LocalLink may cache the last successful browser snapshot for relaunch UX, but live state is always replaced when /api/state succeeds.

State When it appears Why it matters
Up The runtime manager reports a matching running service. The service is externally observable and can show CPU, memory, or uptime when the manager provides it.
Restarting The manager reports startup, restart, launch, stop, or initialization states. The user gets a degraded but truthful status instead of a false success or false outage.
Down The manager is reachable and reports no matching runtime instance for a declared service. Down means LocalLink could verify absence; it is not used when probing failed.
Unknown Docker, PM2, Windows probing, or a trustworthy service mapping is unavailable. This preserves the PRD rule: unknown is better than guessing down.

Workspace scaffold

The CLI generator supports both locallink init and locallink init <workspace-name>. It creates a safe starter workspace without overwriting existing files. If a README already exists, LocalLink writes README.locallink.md instead.

my-local-infra/
├── .env
├── .env.example
├── .gitignore
├── Taskfile.yml
├── docker-compose.yml
├── ecosystem.config.js
├── mcp-registry.json
├── README.md (or README.locallink.md)
└── AGENTS.md
Scaffolded defaults LocalLink writes empty but valid starter configuration stubs for Docker, PM2, Task, registry, env templates, and agent instructions.
Private workspace ready The generated tree is designed to live inside a private infrastructure repo and stay relative-path-safe for WSL / Windows workflows.

AGENTS.md contract

The scaffolded AGENTS.md is the permanent AI context file for the workspace. It captures LocalLink-specific rules that coding agents should follow when adding new services.

Rule What it means
Dependency resolution Services must not hardcode ports or URLs. They should consume upstream connectivity strictly from LocalLink-managed environment variables.
Scaffolding conventions Docker services belong in docker-compose.yml; PM2 or task-backed services belong in ecosystem.config.js; optional metadata should be declared alongside each service.
Network isolation Phase 1 is localhost-only and coerces non-loopback bind hosts back to 127.0.0.1. Phase 2 edge features are optional, detectable, and can be disabled from .env.

Dockerfile as the standard local-service interface

The updated PRD requires native Node and Python services to expose a declarative Dockerfile blueprint, even when the service is executed directly under PM2 instead of being built as a live image. LocalLink treats the Dockerfile as a static configuration contract.

Dockerfile instruction How LocalLink uses it
EXPOSE Feeds the declared port into the service metadata and port-display logic when no runtime port is already resolved.
ENV Extends the service’s declared environment-variable metadata so the dashboard and agents can see what the service expects.
CMD / ENTRYPOINT Provides the static launch contract that LocalLink can expose to the dashboard and the blueprint compliance checker.

For multi-service repos, LocalLink supports an explicit locallink.dockerfile metadata path in ecosystem.config.js, so each PM2 service can point to its own blueprint file.

Service metadata model

Services can now expose richer context to both humans and agents. The dashboard renders this metadata directly on service cards.

Field Compose label Ecosystem metadata Purpose
Dependencies locallink.dependsOn dependsOn Which upstream services this component needs before it is useful.
Downstream locallink.downstream downstream Which other services or user-facing surfaces rely on this component.
Env vars locallink.envVars envVars Operational environment variables worth checking during debugging or setup.
Docs locallink.docsUrl docsUrl Canonical external reference for the runtime or service.
Dockerfile blueprint n/a (compose services already live in Docker) dockerfile Explicit path to the Dockerfile blueprint for native PM2 / task-backed services. 
// ecosystem.config.js
locallink: {
  name: 'Queue Worker',
  group: 'pm2',
  runtime: 'pm2',
  dockerfile: './blueprints/queue-worker.Dockerfile',
  portEnv: 'QUEUE_WORKER_PORT',
  dependsOn: ['Postgres Compose'],
  downstream: ['LocalLink Dashboard UI'],
  envVars: ['QUEUE_WORKER_PORT', 'POSTGRES_PORT'],
  docsUrl: 'https://pm2.keymetrics.io/docs/usage/application-declaration/',
}

Resource watch

The Resource watch view surfaces the top local processes, flags abnormal consumers, and lets the user inspect or terminate them without leaving the dashboard.

Capability Behavior
Top-process summary Tracks the current top CPU process, top RAM process, total processes in the table, and how many crossed LocalLink’s warning thresholds.
Inspect action Loads process details like PID, parent PID, command, CPU, RAM, uptime, and start time.
Terminate action Sends SIGTERM or SIGKILL through the local-only HTTP API, with safety guards that prevent killing PID 1 or LocalLink itself.

Operating LocalLink

Daily local workflow Run locallink doctor when setup feels wrong, locallink web for the dashboard, and locallink snapshot when you need a JSON view for debugging or agent handoff.
Adding a PM2 service Add the app to ecosystem.config.js, include locallink metadata, point dockerfile at its blueprint, and use env variables instead of hardcoded ports or URLs.
Adding a Docker service Add the service to docker-compose.yml, declare ports through env variables, and add locallink.* labels for display name, group, dependencies, env vars, and docs.
Adding a Taskfile service Add an ecosystem entry with runtime: 'taskfile' and taskName, then provide tasks like name:start, name:stop, name:restart, or name:up.

Lifecycle actions are declaration-first. LocalLink only runs actions for services declared in docker-compose.yml or ecosystem.config.js, which keeps agent-triggered operations bounded to the workspace contract.

Phase 2 advisor

Phase 2 remains optional. LocalLink stays local-only by default, but the advisor can detect a usable edge path and surface it in the dashboard.

Tailscale detection When tailscale status --json succeeds, LocalLink can recommend a private network edge with the detected tailnet and IP details.
Pocket ID identity Pocket ID is an optional application identity provider. A new workspace remains extension-free until identity is declared. Keep Pocket ID private behind Tailscale Serve and register internal services through a generic OIDC contract. Open the private SSO guide.
Reverse-proxy detection LocalLink checks for Caddy, Traefik, and Nginx and recommends a reverse-proxy edge when those tools are present. 
LOCALLINK_ENABLE_PHASE2_ADVISOR=true
LOCALLINK_PHASE2_PREFERRED_EDGE=auto
POCKET_ID_APP_URL=https://pocket-id-host.your-tailnet.ts.net
POCKET_ID_PORT=1411

See How LocalLink out-of-box extensions work for the declaration, readiness, runtime, and secret boundaries.

Troubleshooting

Symptom Likely cause What to do
Services show Unknown Docker, PM2, Windows probing, or service mapping was unavailable. Run locallink doctor, verify external tools are installed, and check service names / runtimeName metadata.
Lifecycle action returns unavailable runtime The required external binary is missing from PATH. Install Docker, PM2, or Task as needed, then rerun locallink doctor.
Blueprint compliance warning A PM2 or task-backed service lacks a readable Dockerfile blueprint. Add locallink.dockerfile in ecosystem.config.js and include EXPOSE, ENV, and CMD / ENTRYPOINT where useful.
Dashboard will not start The configured host/port is unavailable or invalid. Check LOCALLINK_BIND_HOST and LOCALLINK_WEB_PORT in .env, or stop the process already using that port.
LocalLink is intentionally local-first. The dashboard stays on loopback by default, the MCP surface stays on stdio, and all Phase 2 edge recommendations remain optional.
